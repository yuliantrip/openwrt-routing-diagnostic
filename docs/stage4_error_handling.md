# Этап 4 — Обработка ошибок и деградация без потери полезности

## Цель этапа
Задать детерминированное поведение `owrt-diag.sh` при проблемах окружения, сохранив принципы:
- **READ-ONLY**;
- **максимум полезных данных даже при частичном отказе**;
- **понятный отчёт о том, что не удалось собрать и почему**.

## 1) Классы ошибок

1. **E_SPACE** — недостаточно места в целевом каталоге (`/tmp` по умолчанию).
2. **E_CMD_MISSING** — отсутствует утилита/бинарник.
3. **E_PERM** — ошибка доступа (permission denied, недоступен файл/сокет).
4. **E_TIMEOUT** — команда зависла/выполнилась слишком долго.
5. **E_EXEC** — команда завершилась с кодом != 0.
6. **E_IO** — ошибка записи артефакта (readonly fs, no space left on device).
7. **E_PARSE** — неожиданная/пустая структура вывода команды.

## 2) Общий контракт обработки ошибок

### 2.1 Никогда не прерывать сбор целиком
- Ошибка в одной секции не останавливает следующие.
- Каждый сбой пишется в `meta/91_errors.log` + учитывается в `meta/93_collection_matrix.csv`.

### 2.2 Стандартизированный статус секций
- `OK` — команда успешна и данные сохранены.
- `WARN` — частичная деградация (например, команда есть, но вернула пусто).
- `FAIL` — команда не выполнена (missing/perm/timeout/io).
- `SKIP` — команда пропущена из-за режима деградации.

### 2.3 Политика возврата (exit code скрипта)
- `0` — все критичные секции собраны без FAIL.
- `1` — есть WARN, но критичные секции собраны.
- `2` — есть FAIL в критичных секциях.

Критичными считаются:
- `interfaces`, `routing`, `nft/fw4`, `ip rule`, `dns runtime`, `process/sockets`, `errors manifest`.

## 3) Нехватка места в `/tmp` (E_SPACE)

## 3.1 Preflight-оценка
Перед стартом:
- `df -k ${OUT_PARENT}`
- `du -sk` по уже существующей сессии (если продолжение)

Пороги (настраиваемые переменными):
- `MIN_FREE_KB_HARD=8192` (8 MB): ниже — аварийный отказ перед стартом.
- `MIN_FREE_KB_SOFT=32768` (32 MB): ниже — запускается режим «экономии».

### 3.2 Режим «экономии»
При soft-пороге:
- уменьшить хвост логов (`tail 800 -> 200`);
- отключить тяжёлые дубликаты (`nft -a list ruleset` оставить, но без доп. повторов);
- отключить optional-секции (например, netcheck/tcpdump).

### 3.3 Hard-stop
При hard-пороге:
- создать только `meta/90_manifest.log` и `meta/91_errors.log`;
- записать причину отказа и рекомендации (сменить `--out`, очистить `/tmp`);
- завершить с кодом `2`.

## 4) Отсутствие утилит (E_CMD_MISSING)

Для каждой команды через `command -v`:
- если отсутствует — секция `FAIL` или `SKIP` (зависит от критичности);
- в `92_recommendations.log` записать рекомендуемый пакет (если известен);
- продолжить сбор альтернативными командами.

Примеры fallback:
- нет `nft` → всё равно собрать `fw4 print`, `iptables-save` (если есть), `uci export firewall`;
- нет `ss` → fallback на `netstat -lnptu` (если доступен);
- нет `tailscale` → не собирать `tailscale status`, но сохранить процессы/маршруты.

## 5) Ошибки доступа (E_PERM)

- stdout/stderr пишутся в соответствующий файл секции + `91_errors.log`.
- секция помечается `WARN` или `FAIL`.
- повторный запуск той же команды не делается бесконечно (не более 1 ретрая).

## 6) Таймауты (E_TIMEOUT)

- Каждая «тяжёлая» команда запускается через таймаут (например, `timeout 10s` или busybox-эквивалент).
- При таймауте:
  - фиксируем `RC=124` (или эквивалент);
  - записываем `E_TIMEOUT`;
  - продолжаем следующую секцию.

Рекомендуемые лимиты:
- `nft list ruleset`: 15s
- `fw4 print`: 10s
- `logread` фильтры: 10s
- `tailscale netcheck`: 12s

## 7) Ошибки записи (E_IO)

- При невозможности писать в `raw/anon/meta`:
  1. попытка fallback в `/tmp/owrt-diagnostic-fallback`;
  2. если и там ошибка — вывести краткий stderr в консоль и завершить `2`.

## 8) Контроль целостности артефактов

После каждой секции:
- проверить, что файл создан и не пуст (где это ожидается);
- записать размер файла и sha256 (если есть `sha256sum`) в `90_manifest.log`.

Если файл пустой там, где ожидался вывод:
- статус `WARN` + код `E_PARSE`.

## 9) Анонимизация и отказоустойчивость

- Ошибки анонимизации не должны удалять raw-данные.
- Если `mask_sensitive_data` упал:
  - сохранить raw;
  - отметить anon-файл как `FAIL (E_EXEC)`;
  - добавить предупреждение о ручной редакции перед публикацией.

## 10) Минимальный гарантированный набор даже при деградации

Даже в degraded-режиме скрипт пытается сохранить:
1. `11_interfaces_runtime.log`
2. `12_routing_runtime.log`
3. `13_fw4_nft_runtime.log` (или fallback)
4. `15_dns_runtime.log`
5. `16_clash_tailscale_runtime.log`
6. `meta/90_manifest.log`
7. `meta/91_errors.log`

## 11) Событийный журнал скрипта

Добавить `meta/95_runtime_events.log`:
- старт/стоп секций;
- переход в degraded-режим;
- причины `SKIP`;
- итоговый summary по статусам `OK/WARN/FAIL/SKIP`.

## 12) Готовность к Этапу 5

На Этапе 5 реализуем эти правила непосредственно в Bash-коде:
- `run_and_capture()` с timeout/retry/status;
- централизованный `record_error()`;
- `preflight_space_guard()`;
- fallback-пути записи;
- итоговый код возврата по критичности секций.
