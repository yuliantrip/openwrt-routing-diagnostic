# Merge Playbook (для новичка): как закрыть конфликты в PR

Ниже — максимально практичный сценарий именно для вашего случая, когда конфликтуют `owrt-diag.sh` и `README.md`.

## Вариант A (самый простой): через GitHub веб-интерфейс

### Шаг 1. Откройте PR и нажмите **Resolve conflicts**

Если видите конфликт-блок:

```text
<<<<<<< codex/...
...ваша новая версия...
=======
...версия из main...
>>>>>>> main
```

### Шаг 2. Для `owrt-diag.sh` оставьте версию из `codex/...`

- Оставить верхний блок (из вашей ветки PR).
- Удалить нижний блок из `main`.
- Удалить маркеры `<<<<<<<`, `=======`, `>>>>>>>`.

### Шаг 3. Для `README.md`

- Если хотите текущий актуальный README из ветки PR — делайте то же самое (оставляйте блок `codex/...`).
- Если в `main` есть важные новые строки, добавьте их вручную в итоговый файл.

### Шаг 4. Сохранение

- Нажмите **Mark as resolved** для каждого файла.
- Нажмите **Commit merge**.

После этого GitHub пересчитает PR и обычно уберёт статус конфликтов.

---

## Вариант B (рекомендую): через командную строку (без ручного редактирования кучи блоков)

### Шаг 1. Перейти в ветку PR

```bash
git checkout <ваша_pr_ветка>
```

### Шаг 2. Подтянуть `main` и слить в вашу ветку

```bash
git fetch origin
git merge origin/main
```

### Шаг 3. Для конфликтных файлов оставить вашу версию (ветка PR)

```bash
git checkout --ours owrt-diag.sh README.md
```

> В этом контексте `--ours` = текущая PR-ветка.

### Шаг 4. Зафиксировать merge

```bash
git add owrt-diag.sh README.md
git commit -m "resolve conflicts: keep PR branch versions of owrt-diag.sh and README"
git push
```

После push конфликт в PR должен исчезнуть.

---

## Быстрые проверки перед Merge

Проверьте, что в файлах нет конфликт-маркеров:

```bash
rg "^<<<<<<<|^=======|^>>>>>>>" owrt-diag.sh README.md
```

Если команда ничего не вывела — всё чисто.

Проверьте синтаксис скрипта:

```bash
sh -n owrt-diag.sh
```

---

## Что выбрать из кнопок GitHub: current/upcoming/both?

Кнопки могут путать. Надёжнее ориентироваться по меткам внутри блока:
- если хотите оставить вашу новую реализацию, выбирайте часть под `<<<<<<< codex/...`;
- не выбирайте автоматически `both` для shell-скрипта, часто это оставляет конфликтный мусор/дубли.
